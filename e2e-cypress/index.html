Why test end-to-end?

We test...

  • to ensure our application works within the target eco-system.
  • automated to get quick and reliable feedback.
  • to detect breaking features as soon as possible.
  • to ensure the best quality.

Cypress as an alternative

"Cypress is a next generation front end testing tool built for the modern web. We address the key pain points developers and QA engineers face when testing modern applications."

Cypress in a nutshell

It is just another node module...


npm install cypress --save-dev

...which comes with everything you need to start testing!

You can visit your app...


cy.visit('/movies');

Cypress docs

...interact with elements...


// click on a fancy button
cy.get('[data-cy="my-fancy-button"]').click();
// clear and type into a fancy input
cy.get('[data-cy="my-fancy-input"]').clear().type('Important stuff');

Cypress docs

...make assertions...


// check whether the input value is something
cy.get('[data-cy="my-fancy-input"]').should('have.value', 'Important stuff');

Cypress docs

...and even intercept backend calls!


// use a static response
cy.intercept({ method: 'GET', url: '/services/rest/movies' }, [...]);
// wait for a request before moving on
cy.intercept({ method: 'GET', url: '/services/rest/movies' }).as('getMovies');
cy.wait('@getMovies');
// intercept a call and work with the response
let movies;
cy.intercept({ method: 'GET', url: '/services/rest/movies' }, (request) => {
  request.continue((response) => {
    movies = response.body;
  });
}).as('getMovies');
cy.wait('@getMovies').then(() => {
  // we can work with movies only when the request is done
});

  beforeEach(() => {
    moviesPage = new MoviesPage();
  });

  it('should create a new movie', () => {
    // visit the movie overview
    cy.visit('/movies'):

    // do stuff
    moviesPage.clickAddButton();
    // do something to add a new movie...
  });
});

Some best practices:

  • using the PageObject pattern is always a good idea!
  • use data-* attributes to uniquely identify elements.
  • do not store the result of a chainable command in a variable.
  • tests should not depend on another.
  • use as many assertions as needed. It's not unit testing.
